내가 Copilot을 안 썼는데, 왜 커밋에 Copilot 이름이 들어가 있죠?

• VS Code가 Co-Authored-by: GitHub Copilot 태그를 커밋 메시지에 자동 삽입하는 현상이 보고됐어요. Copilot을 실제로 사용하지 않은 코드에도 붙는다는 점에서 커뮤니티 반응이 뜨거웠죠.
• VS Code는 Code - OSS라는 오픈소스 저장소와, Microsoft가 별도 라이선스로 배포하는 Visual Studio Code 바이너리가 따로 존재해요. 이 구조 때문에 오픈소스에 없는 동작이 배포판에 슬그머니 들어오기도 해요.
• 팀 단위로 커밋 히스토리를 관리하거나 오픈소스 기여를 할 때, 내 의도와 다른 메타데이터가 붙는 건 꽤 민감한 문제예요. git log나 git commit --amend로 이미 들어간 태그를 정리해야 할 수도 있어요.

💡 오늘의 인사이트

에디터를 그냥 믿고 쓰다 보면 이런 일이 생겨요. VS Code 설정에서 Copilot 관련 항목을 한 번쯤 훑어보는 게 좋을 것 같아요. 특히 오픈소스 프로젝트에 기여하거나, 회사 정책상 AI 도구 사용 이력을 관리해야 한다면 더더욱요. 도구가 내 대신 결정을 내리게 두지 말고, 내가 도구를 이해하고 쓰는 게 결국 더 편해요.

애플 워치 화면에 지도를 넣는 데 6년이 걸린 이유

• Pedometer++ 개발자는 초기에 서버에서 지도 이미지를 생성해 내려받는 방식을 썼어요. 그러다 오프라인 지원과 성능 문제를 해결하려고 2021년에 아예 SwiftUI 기반 커스텀 지도 렌더링 엔진을 직접 만들었죠.
• MapKit을 안 쓴 이유가 흥미로워요. 다크 모드 제한, 애니메이션 자유도, 그리고 등고선이나 트레일 같은 지형 정보 커버리지가 부족했기 때문이에요. 서드파티 타일 데이터를 직접 붙이는 게 오히려 나았던 거죠.
• 작은 화면에서 지도와 운동 지표를 동시에 보여주는 레이아웃은 수없이 많은 반복 설계 끝에, 결국 사용자가 직접 어떤 지표를 볼지 고를 수 있는 설정 가능한 인터페이스로 마무리됐어요.

💡 오늘의 인사이트

플랫폼이 제공하는 기본 컴포넌트를 쓰면 빠르게 시작할 수 있지만, 그게 항상 정답은 아니에요. 이 개발자는 6년 동안 직접 만든 엔진을 유지보수하는 비용을 감수하면서도 커스텀 솔루션을 선택했어요. 내 서비스의 핵심 경험이 플랫폼 기본값에 종속되는 게 불편하다면, 직접 만드는 선택지도 진지하게 고려해볼 만해요. 단, 그 유지보수 비용까지 계산에 넣고요.

브라우저를 처음부터 다시 만드는 사람들, 4월에 이런 걸 해냈어요

• 독립 브라우저 프로젝트 Ladybird가 4월 한 달 동안 333개의 PR을 머지했어요. pdf.js를 활용한 인라인 PDF 뷰어, SQLite 기반 방문 기록과 주소창 자동완성까지 실제로 쓸 수 있는 기능들이 착착 붙고 있죠.
• 성능 개선도 눈에 띄어요. 자바스크립트 바이트코드 생성을 백그라운드 스레드로 옮겨서 YouTube 로딩 시 메인 스레드 부하를 약 200ms 줄였고, ARM64 어셈블리를 직접 튜닝해서 JS 함수 호출 속도를 높였어요. iframe도 별도 스레드에서 렌더링하도록 바꿨고요.
• 보안 측면에서는 WebAssembly.Memory를 공유할 때 TypedArray 뷰에서 발생하는 use-after-free 취약점을 발견하고 수정했어요. 브라우저를 바닥부터 만들면서도 보안 이슈를 꼼꼼히 챙기고 있다는 게 인상적이에요.

💡 오늘의 인사이트

크롬이나 사파리가 당연하게 느껴지는 세상에서, 브라우저를 처음부터 다시 만드는 팀이 있다는 게 신기하죠. 이 프로젝트를 보면서 느끼는 건, 거대한 시스템도 결국 작은 PR 하나하나가 쌓인 결과라는 거예요. 내가 오늘 올린 PR이 별거 아닌 것 같아도, 333개 중 하나가 되는 거니까요.
